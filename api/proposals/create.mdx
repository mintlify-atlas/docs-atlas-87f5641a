---
title: "Create Proposal"
api: "POST /api/proposals"
description: "Create a new proposal. Only one draft proposal is allowed per user. If a draft already exists, it will be returned instead of creating a new one."
---

## Authentication

This endpoint requires authentication via Bearer token.

```bash
Authorization: Bearer <token>
```

## Request Body

<ParamField body="title" type="string" required>
  The title of the proposal.
</ParamField>

<ParamField body="description" type="string" default="">
  Optional description of the proposal.
</ParamField>

<ParamField body="template_id" type="string">
  Optional template ID to use for the proposal.
</ParamField>

## Response

<ResponseField name="proposal" type="object">
  The created or existing proposal object.
  
  <Expandable title="Proposal Object">
    <ResponseField name="id" type="string">
      Unique proposal identifier (UUID).
    </ResponseField>
    
    <ResponseField name="proposalCode" type="string">
      Human-readable proposal code (format: PROP-YYYYMMDD-XXXX).
    </ResponseField>
    
    <ResponseField name="title" type="string">
      The title of the proposal.
    </ResponseField>
    
    <ResponseField name="description" type="string">
      The description of the proposal.
    </ResponseField>
    
    <ResponseField name="template_id" type="string">
      The template ID if specified.
    </ResponseField>
    
    <ResponseField name="status" type="string">
      Current status of the proposal (e.g., "draft").
    </ResponseField>
    
    <ResponseField name="created_at" type="string">
      ISO 8601 timestamp of when the proposal was created.
    </ResponseField>
    
    <ResponseField name="updated_at" type="string">
      ISO 8601 timestamp of the last update.
    </ResponseField>
    
    <ResponseField name="user_id" type="string">
      The ID of the user who owns the proposal.
    </ResponseField>
    
    <ResponseField name="user_email" type="string">
      The email of the user who owns the proposal.
    </ResponseField>
    
    <ResponseField name="user_name" type="string">
      The name of the user who owns the proposal.
    </ResponseField>
    
    <ResponseField name="uploaded_files" type="object">
      Dictionary of uploaded files organized by category.
    </ResponseField>
    
    <ResponseField name="text_inputs" type="object">
      Dictionary of text inputs provided by the user.
    </ResponseField>
    
    <ResponseField name="metadata" type="object">
      Additional metadata including analysis results and status flags.
    </ResponseField>
    
    <ResponseField name="completed_steps" type="array">
      Array of step numbers that have been completed (e.g., [1, 2]).
    </ResponseField>
    
    <ResponseField name="step_completion" type="object">
      Detailed completion status for each step with substatus information.
    </ResponseField>
  </Expandable>
</ResponseField>

<ResponseField name="message" type="string">
  Optional message, present when returning an existing draft ("Returning existing draft proposal").
</ResponseField>

## Example Request

```bash cURL
curl -X POST https://api.igadregion.org/api/proposals \
  -H "Authorization: Bearer YOUR_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "title": "Climate Resilience in East Africa",
    "description": "Proposal for improving climate adaptation strategies"
  }'
```

## Notes

- **One Draft Per User**: The system enforces a single draft proposal per user. If you call this endpoint when a draft already exists, you'll receive the existing draft instead of creating a new one.
- **Proposal Code**: The `proposalCode` is automatically generated in the format `PROP-YYYYMMDD-XXXX` where YYYYMMDD is the creation date and XXXX is a random suffix.
- **Step Completion**: The `completed_steps` array and `step_completion` object are computed based on actual data presence and provide a persistent tracking of proposal progress.
- **User Information**: The authenticated user's ID, email, and name are automatically attached to the proposal.
